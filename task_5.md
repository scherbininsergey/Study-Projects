# Чек-лист Grey-box проверок приложения «Найди и иди»

## 1. Процесс сравнения и доработки

| Этап | Описание | Результат |
|------|----------|-----------|
| **Мой черновик** | Сфокусирован на проверке согласованности UI-поведения с внутренним состоянием БД, API-пейлоадами, кэшем и логами. Явно выделены проверки целостности данных, TTL сессий, SQL-валидации и механизмов фолбэков. | Высокая глубина покрытия внутренних состояний, но местами избыточен для базового Grey-box. |
| **Черновик ИИ (1-я итерация)** | Сгенерирован по промпту «Сделай Grey-box чек-лист для SC21-01..06». ИИ не учёл доступ к БД: проверки свелись к Black-box + абстрактные фразы вроде «проверить сохранение в БД» без указания таблиц, полей или логики согласования. | Низкая практическая ценность для Grey-box. Требует конкретизации. |
| **Уточнение промпта** | `«Сгенерируй Grey-box чек-лист для SC21-01..SC21-06. У тестировщика есть READ-доступ к PostgreSQL/SQLite БД с таблицами: shops (id, name, lat, lon, category_id, is_active), search_history (query, created_at, user_id), route_sessions (id, start_coords, end_coords, status, created_at, expires_at). Для каждого пункта укажи: 1) действие в UI, 2) ожидаемое поведение на экране, 3) конкретную проверку в БД/API/логах (какие таблицы/поля сверять, какие запросы эмулировать, как верифицировать синхронизацию кэша и БД). Избегай абстракций.»` | ИИ выдал структурированный вариант с явными таблицами, проверками согласования и верификацией состояний. |
| **Итог сравнения** | Мой вариант сильнее в проверке граничных условий и конфликтов ТЗ. Вариант ИИ (после уточнения) чётче трассирует БД-сущности и проще масштабируется для автоматизации SQL-проверок. | Финальный чек-лист объединяет строгую трассировку к БД (ИИ) с валидацией противоречий ТЗ, fallback-логикой и негативными сценариями (мой вариант). |

## 2. Финальный Grey-box чек-лист проверок

### 🔹 SC21-01: Поле ввода названия магазина
- [ ] **Ввод ≤50 валидных символов** → UI: кнопка активна, поиск выполняется. **БД/API:** запрос `SELECT * FROM shops WHERE name ILIKE '%input%'` возвращает те же записи, что и маркеры на карте. Проверка absence of SQL-инъекций в логах API.
- [ ] **Ввод >max_length (например, 51 символ)** → UI: обрезка или ошибка. **БД:** поле `shops.name` имеет ограничение `VARCHAR(50)`. Проверка, что UI не отправляет payload, вызывающий `413/400` или усечение на стороне БД без предупреждения.
- [ ] **Строки из пробелов/спецсимволов** → UI: валидация блокирует поиск. **БД/API:** проверка, что бэкенд не кэширует пустые запросы в `search_history`.
- [ ] **Сохранение истории поиска** → UI: выпадающий список при фокусе. **БД:** запрос к `search_history` возвращает записи в обратном хронологическом порядке. Проверка дедупликации (одинаковые запросы не плодят строки) и лимита (например, `LIMIT 10`).
- [ ] **Выбор из истории** → UI: автозапуск поиска. **БД/API:** сверка `search_history.query` с фактическим payload, уходящим на поиск. Проверка отсутствия race-condition при быстром выборе нескольких записей.

### 🔹 SC21-02: Поле выбора типа магазина
- [ ] **Отображение списка категорий** → UI: выпадающее меню. **БД:** `SELECT id, name FROM categories WHERE is_active = true` полностью совпадает со списком в UI. Проверка отсутствия скрытых/заархивированных категорий.
- [ ] **Выбор категории** → UI: плейсхолдер заменяется на название. **БД/API:** в payload поиска корректно передаётся `category_id`, а не текстовое значение.
- [ ] **Изменение справочника в БД** → UI: при следующем открытии/обновлении список синхронизирован. **Проверка:** наличие механизма cache-invalidation или TTL у локального кэша категорий.

### 🔹 SC21-03: Кнопка «Поиск» и область результатов
- [ ] **Активация кнопки** → UI: активна при ≥2 символах ИЛИ выбранной категории. **БД/API:** проверка, что при несоответствии условий запрос не уходит (отсутствие записей в `api_requests_log` или status `400`).
- [ ] **Успешный поиск** → UI: маркеры на карте. **БД:** сверка количества маркеров с `COUNT(*)` из запроса поиска. Проверка координат (`lat`, `lon`) на совпадение с данными в `shops`.
- [ ] **Пустой результат** → UI: сообщение «Ничего не найдено». **БД/API:** ответ `200 OK` с пустым массивом или `204`. Проверка отсутствия ошибок в серверных логах.
- [ ] **Ошибки сети/сервера** → UI: тост/экран ошибки. **БД/API:** логирование в `error_logs` с кодом (5xx, timeout, SSL). Проверка отсутствия «потерянных» сессий поиска в `search_history`.
- [ ] **Кластеризация/лимит маркеров** → UI: при N>50 маркеры группируются. **БД:** проверка, что бэкенд возвращает пагинированный набор или флаг `has_more`, а UI не обрезает данные молча.

### 🔹 SC21-04 & SC21-05: Построение и отмена маршрута
- [ ] **Активация кнопки маршрута** → UI: активна при выбранном флаге ИЛИ категории. **БД:** проверка, что `end_coords` берётся из `shops.lat/lon` выбранного объекта или результата `MIN(ST_Distance(...))` для категории.
- [ ] **Построение маршрута** → UI: линия на карте, кнопка меняет стиль/текст. **БД:** запись в `route_sessions` со статусом `active`, корректными `start_coords` (текущая геолокация) и `end_coords`. Проверка отсутствия дублирующих сессий для одного пользователя.
- [ ] **Отмена маршрута** → UI: линия скрывается, кнопка возвращается в исходное состояние. **БД:** статус сессии меняется на `cancelled` или запись помечается `is_deleted=true`. Проверка, что `expires_at` не сбрасывается некорректно.
- [ ] **Перезапуск приложения с активным маршрутом** → UI: маршрут восстанавливается. **БД:** проверка, что приложение запрашивает последнюю сессию с `status='active'` и `expires_at > NOW()`. Сверка координат с БД.
- [ ] **Проверка ограничения «только кнопка Отмена»** → UI: жест «Назад»/сворачивание не отменяют маршрут. **БД/State:** проверка, что сессия не завершается при системных событиях. Логирование в `app_state_logs` подтверждает сохранение статуса `active`.

### 🔹 SC21-06: Общие требования и системные взаимодействия
- [ ] **Точная геолокация vs низкий заряд** → UI: отключение GPS при <15%. **БД/Logs:** проверка, что `location_source` в логах меняется с `GPS` на `NETWORK/WIFI`. Координаты в `route_sessions` обновляются по последним валидным данным.
- [ ] **Восстановление геолокации** → UI: возврат к точному позиционированию при заряде >15%. **БД:** проверка, что `start_coords` в активных сессиях корректно обновляются без разрыва маршрута.
- [ ] **TTL и очистка данных** → UI: история и маршруты не растут бесконечно. **БД:** проверка наличия cron-задачи или soft-delete логики для `search_history` (старше 30 дней) и `route_sessions` (старше 7 дней). Сверка с политикой хранения.
- [ ] **Конфликт состояний при быстром поиске** → UI: новый поиск не «ломает» отображение предыдущего. **БД:** проверка, что concurrent-запросы обрабатываются атомарно, а кэш результатов не возвращает stale-данные.

## 3. Примечания к Grey-box методике

1. **Доступ к БД используется только для верификации**, а не для прямого изменения данных. Все проверки выполняются в режиме `READ ONLY` или на тестовом стенде с изолированной БД.
2. **Сверка UI ↔ БД** является ядром Grey-box: маркеры, история, статусы маршрутов и координаты должны иметь 1-к-1 соответствие между отображением и хранилищем.
3. **Противоречия ТЗ** (SC21-01 vs SC21-03, SC21-04 vs SC21-05) проверяются через наблюдение за состоянием сессий в `route_sessions` и логами API: если UI разрешает действие, которое БД/бэкенд блокирует, это фиксируется как дефект согласования слоёв.
4. **Кэш и синхронизация** проверяются явно: локальное хранение истории/категорий должно иметь чёткий механизм инвалидации при изменении справочников в БД.
5. Чек-лист готов к конвертации в автоматизированные SQL-проверки (например, через `pytest + SQLAlchemy`), интеграции с API-тестами (Postman/Newman) и ручному Grey-box прогону с доступом к админ-панели/БД.
