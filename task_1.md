# Задание 1. Классификация тестирования

| Артефакт | По уровням тестирования (Что по границам?) | По стадии процесса (Когда/Зачем?) | По знанию устройства системы | По направленности сценария | По выполнению кода |
|----------|-------------------------------------------|---------------------------------|-----------------------------|---------------------------|-------------------|
| **1. Модуль перевода средств**<br>(изолированный, без реального шлюза) | **Компонентное тестирование**<br>Проверка логики модуля «в вакууме»: корректность расчёта комиссии, валидация входных параметров, обработка ответов заглушки (мока). | **Пре-интеграционная фаза**<br>Валидация контракта модуля до встраивания в приложение. Цель: выявить дефекты логики и обработки данных, пока отладка быстрая и изолированная. | **Метод серого ящика**<br>Тестирование с доступом к логам модуля, тестовой базе данных и конфигурации моков. Видим структуру запросов и статусы транзакций, но не анализируем исходный код. | **Позитивное и негативное**<br>• Позитивное: перевод с корректными реквизитами и суммой в пределах лимита.<br>• Негативное: отрицательная сумма, несуществующий счёт, таймаут или ошибка 5хх от мока, проверка идемпотентности (повторный запрос не списывает средства дважды). | **Динамическое тестирование**<br>Запуск модуля в изоляции через модульные или API-тесты. Подача тестовых данных, получение ответов от заглушки, валидация статусов и записей в тестовой базе данных в реальном времени. |
| **2. Макет интерфейса**<br>(основные экраны, Figma/Sketch) | **— (Статический артефакт)**<br>Пирамида уровней применима только к динамическому тестированию исполняемого кода. Для макетов используются методы статического тестирования (ревью дизайна), которые находятся вне этой иерархии. | **Фаза валидации и утверждения дизайна**<br>Включающая ревью дизайна: проверка полноты состояний экранов, логики переходов и соответствия техническому заданию. Цель: получить подтверждение от продуктовой команды и устранить проблемы юзабилити до начала разработки. | **Метод чёрного ящика**<br>Визуальная оценка без доступа к коду. Сравниваем видимый интерфейс с требованиями, дизайн-системой и гайдлайнами. Внутренняя реализация недоступна по определению. | **Позитивное и негативное (на уровне дизайна)**<br>• Позитивное: соответствие дизайн-системе (отступы, цвета, типографика, визуальная иерархия).<br>• Негативное: проверка наличия в макете состояний для ошибок валидации, пустых списков, переполнения текстом, тёмной темы и адаптации под разные разрешения экрана. | **Статическое тестирование**<br>Анализ артефакта без выполнения кода. Проверка визуальных параметров, полноты состояний и логики переходов через ревью в Figma, плагины для проверки контрастности и доступности. |
| **3. Первая полная сборка**<br>(для внутренних сотрудников, `.apk`/`.ipa`) | **Системное тестирование**<br>Проверка приложения как единого целого: интеграция модуля перевода с интерфейсом, навигацией, хранилищем и заглушкой шлюза. Валидация сквозных пользовательских сценариев и обработки системных событий. | **Альфа-тестирование**<br>Закрытая проверка внутренними сотрудниками компании до публичного релиза. Цель: выявить критические дефекты, оценить стабильность базовых сценариев и подтвердить готовность продукта к бета-тестированию. | **Метод чёрного ящика + метод серого ящика**<br>• Чёрный ящик: основная проверка через пользовательский интерфейс, как у конечного пользователя.<br>• Серый ящик: при локализации дефектов используется доступ к логам, анализу сетевого трафика (DevTools/Charles) и тестовой базе данных. | **Позитивное, негативное и исследовательское**<br>• Позитивное: полный пользовательский путь от авторизации до получения чека о переводе.<br>• Негативное: обрыв сети, прерывание операции сворачиванием приложения, превышение лимитов, невалидные входные данные.<br>• Исследовательское: свободное тестирование на основе опыта использования аналогичных банковских приложений — нестандартные последовательности действий, параллельные процессы, скрытые дефекты бизнес-логики. | **Динамическое тестирование**<br>Работа с запущенным приложением. Подача тестовых данных через интерфейс, анализ ответов, чтение логов, проверка состояния тестовой базы данных и поведения системы в реальном времени при взаимодействии пользователя с интерфейсом. |

## Примечания к таблице

1.  **Независимость осей классификации**: Каждая колонка отвечает на свой вопрос. Один и тот же артефакт проверяется одновременно по всем осям.

2.  **Пересечение категорий**: Например, для Артефакта 3: `Системное тестирование` (уровень) проводится на стадии `Альфа-тестирование` (процесс), преимущественно `методом чёрного ящика` (знание), с применением `позитивных/негативных/исследовательских` сценариев (направленность), в режиме `динамического тестирования`.

3.  **Гибкость значений**: В ячейках указаны доминирующие подходы. В реальной практике один артефакт может тестироваться разными методами в зависимости от цели конкретной проверки.

4.  **Соответствие стандарту**: Терминология приведена к стандарту **ГОСТ Р 56940-2016** (адаптация ISTQB): `Уровни тестирования`, `Типы тестирования`, `Методы тестирования`, `Статическое/Динамическое тестирование`.

5.  **Примечание по уровням для статических артефактов**: Пирамида уровней тестирования (Компонентное → Интеграционное → Системное → Приёмочное) применима только к динамическому тестированию исполняемого кода. Для статических артефактов (требования, макеты, документация) используются методы статического тестирования (ревью, инспекция, статический анализ), которые находятся вне этой иерархии.

## Глоссарий ключевых терминов

| Термин | Определение |
|--------|-------------|
| **Заглушка (Mock/Stub)** | Искусственная имитация внешнего сервиса или компонента, возвращающая предопределённые ответы для изоляции тестируемого модуля. |
| **Идемпотентность** | Свойство операции, при котором повторное выполнение того же запроса не изменяет результат после первого успешного выполнения (критично для финансовых транзакций). |
| **Сигн-офф (Sign-off)** | Формальное подтверждение заинтересованной стороной (продукт, бизнес, заказчик) готовности артефакта или продукта к переходу на следующий этап. |
| **Сниффинг трафика** | Перехват и анализ сетевых запросов/ответов между клиентом и сервером с помощью инструментов (Charles, DevTools, Wireshark) для отладки и локализации дефектов. |
| **Инспекция** | Формализованный вид статического тестирования, проводимый обученным модератором по чек-листу с фиксацией дефектов и метрик. |
