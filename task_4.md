# Чек-лист Black-box проверок приложения «Найди и иди»

## 1. Методология и сравнение вариантов
Черновик чек-листа был составлен мной на основе экспертного опыта тестирования мобильных приложений, а затем сгенерирован ИИ-моделью по тем же требованиям (SC21-01 – SC21-06). Ниже приведено сравнение подходов и обоснование финального выбора.

| Критерий оценки | Мой черновик | Черновик ИИ |
|----------------|--------------|-------------|
| **Покрытие требований** | 100%, но с явным расширением неопределённых зон ТЗ | 100%, строго линейное следование тексту ТЗ |
| **Негативные сценарии** | Включены границы ввода, сетевые ошибки, пустые/загрузочные состояния, конфликтные условия | Отсутствуют или сведены к минимому |
| **Состояния UI/UX** | Детализированы: переходы, фидбеки, ориентация экрана, системные жесты | Упомянуты только явно указанные в ТЗ элементы |
| **Проверяемость** | Каждый пункт содержит наблюдаемое ожидаемое поведение | Часть пунктов требует интерпретации («проверка сохранения») |
| **Соответствие Black-box** | Строгое: только входные данные, действия пользователя и видимые результаты | Строгое, но поверхностное |

**Что лучше в моём варианте:** Глубина проработки граничных условий, явное выделение состояний загрузки/ошибок/пустых экранов, проверка противоречий ТЗ через observable-поведение, учёт мобильных паттернов (сворачивание, kill-app, ориентация).
**Что лучше в варианте ИИ:** Лаконичность, чёткая привязка 1-к-1 к формулировкам ТЗ, удобная структура для быстрой сверки аналитиком, отсутствие «лишних» проверок, выходящих за рамки прямого текста требований.

**Итог:** Финальный чек-лист объединяет строгую трассируемость варианта ИИ с обязательными Black-box-расширениями моего варианта. Все пункты проверяемы без доступа к коду, логам или внутренней архитектуре.

## 2. Финальный чек-лист проверок

### 🔹 SC21-01: Поле ввода названия магазина
- [ ] Плейсхолдер отображается при пустом поле и исчезает при вводе первого символа
- [ ] Ввод 0 символов: кнопка «Поиск» неактивна
- [ ] Ввод 1 символа: кнопка «Поиск» неактивна (проверка согласованности с правилом «мин. 2 символа»)
- [ ] Ввод 2+ символов: кнопка «Поиск» переходит в активное состояние
- [ ] Ввод только пробелов/спецсимволов: проверка реакции валидации и состояния кнопки
- [ ] Удаление символов до 1 или 0: кнопка возвращается в неактивное состояние
- [ ] Отображение и выбор элементов истории поиска при фокусе/тапе на поле
- [ ] Выбор из истории: автоматический запуск поиска с подставленным значением

### 🔹 SC21-02: Поле выбора типа магазина
- [ ] По умолчанию отображается плейсхолдер «Выберите тип магазина»
- [ ] Тап по полю открывает выпадающий список
- [ ] Список содержит все категории, указанные в ТЗ
- [ ] Выбор значения: плейсхолдер заменяется на текст выбранной категории
- [ ] Выбор типа магазина активирует кнопку «Поиск» (даже при пустом поле названия)
- [ ] Повторный тап по выбранному значению: список открывается, выбор сохраняется

### 🔹 SC21-03: Кнопка «Поиск» и область результатов
- [ ] Кнопка «Поиск» активна только при условии: ≥2 символа в названии ИЛИ выбран тип магазина
- [ ] Клик по активной кнопке: отображение индикатора загрузки (спиннер/прогресс)
- [ ] Успешный ответ: на карте появляются флажки-маркеры найденных объектов
- [ ] Пустой результат (0 совпадений): отображение сообщения «Ничего не найдено»
- [ ] Ошибка сети/сервера: отображение системного или кастомного сообщения, возможность повторного запроса
- [ ] Область карты визуально занимает ~50% высоты экрана (проверка в портретной и альбомной ориентации)
- [ ] Жесты скролла и зума карты работают без блокировки приложения
- [ ] Тап по флажку: выделение маркера, обновление UI для перехода к построению маршрута

### 🔹 SC21-04 & SC21-05: Кнопка «Построить маршрут» и логика маршрута
- [ ] Кнопка неактивна, если не выбран флаг на карте и не выбран тип магазина
- [ ] Сценарий 1: выбран конкретный флаг → кнопка активна, нажатие строит маршрут до выбранного объекта
- [ ] Сценарий 2: выбран только тип → кнопка активна, нажатие строит маршрут до ближайшего объекта этого типа
- [ ] При нажатии: фон кнопки меняется на #586d73, текст меняется на «Отменить маршрут»
- [ ] На карте отображается линия маршрута, стартовая и конечная точки
- [ ] Нажатие кнопки «Отменить маршрут»: линия скрывается, кнопка возвращается к состоянию «Построить маршрут», цвет #fff
- [ ] Проверка ограничения «только кнопка Отмена»: жест «Назад», тап вне карты, сворачивание приложения не отменяют маршрут
- [ ] Повторное нажатие «Построить маршрут» при уже активном маршруте: корректная обработка (игнор/перестройка)

### 🔹 SC21-06: Общие требования и системные взаимодействия
- [ ] Сворачивание приложения (Home) с активным маршрутом → разворачивание: маршрут сохраняется в UI
- [ ] Полный перезапуск приложения (kill + launch): отображение ранее незаконченного маршрута
- [ ] Изменение геолокации устройства во время отображения маршрута: проверка реакции (обновление/сохранение точки старта)
- [ ] Включение системного режима энергосбережения / заряд батареи <15%: отключение точной геопозиции
- [ ] При отключении точной геопозиции: карта продолжает работать, маршрут строится по последним известным координатам или сетевой локации (проверка наблюдаемого поведения)
- [ ] Выход из режима энергосбережения / заряд >15%: восстановление точной геолокации без перезапуска приложения

## 3. Примечания к выполнению Black-box тестирования
1. **Все проверки выполняются только через UI и системные наблюдатели** (без доступа к логам, БД, сетевым дамперам).
2. **Противоречия ТЗ** (активация кнопки при 1 символе vs мин. 2 символа; «только кнопка Отмена» vs системные жесты) проверяются явно и фиксируются как отклонения от ожидаемого поведения.
3. **Неявные состояния** (загрузка, ошибки, пустые экраны) добавлены в чек-лист как обязательные для Black-box, поскольку их отсутствие в ТЗ является риском, а не оправданием для пропуска проверок.
4. Чек-лист готов к импорту в TestRail/Zephyr, конвертации в автоматизированные сценарии или использованию как база для ручного регрессионного прогона.
